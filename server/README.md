# Платіжний сервер: WayForPay → одноразове посилання → Telegram-бот (SendPulse)

Розвʼязує задачу: оплата на лендінгу → WayForPay → після **підтвердженої** оплати користувача перекидає в Telegram-бот за **одноразовим** посиланням, яким **не можна поділитися** (перше відкриття «привʼязує» доступ до одного користувача; чуже повторне використання блокується).

## Як це працює

```
Лендінг (кнопка «Відкрити доступ»)
        │  GET /pay
        ▼
Сервер створює замовлення + підпис → форма на WayForPay
        │
        ▼
WayForPay: сторінка оплати
   ├── POST /callback (сервер-до-сервера): перевірка підпису.
   │        Якщо Approved → замовлення "paid" + генерується ОДНОРАЗОВИЙ токен
   └── /return (браузер): сторінка очікування → опитує /order-status
                          → редірект на лендінг ?status=approved&tg=<t.me/bot?start=ТОКЕН>
        │
        ▼
Лендінг: подія Facebook «Purchase» → редірект у бота за одноразовим посиланням
        │
        ▼
Telegram-бот (SendPulse): флоу отримує змінну token і робить HTTP-запит
        GET /validate?token={{token}}&uid={{contact.id}}
        ├── валідний і ПЕРШИЙ раз → "valid" → видача міні-курсу (флоу)
        └── уже використаний → "used" → повідомлення «посилання недійсне»
```

Токен «спалюється» під час першої валідації — тому посилання спрацьовує **рівно один раз**
(строгий режим `ALLOW_REENTRY=false`). Навіть той самий користувач повторно курс не отримає;
щоб дозволити покупцю повертатися — постав `ALLOW_REENTRY=true`.

### Формат посилання на бота (важливо!)

Щоб передати токен усередину флоу, SendPulse вимагає проміжний домен **`tg.pulse.is`**:

```
https://tg.pulse.is/ІМʼЯ_БОТА?start=FLOW_ID&token=ОДНОРАЗОВИЙ_ТОКЕН
```

- `FLOW_ID` — ID флоу видачі курсу (див. нижче), береться зі `SENDPULSE_FLOW_ID`;
- `token` — змінна, яку **треба створити в SendPulse → Audience** (тип: текст), саме туди
  збережеться значення при старті;
- сервер будує це посилання автоматично.

## 1. Розгортання сервера

Підійде будь-який хост із Node.js 18+ (Render, Railway, Fly.io, VPS).

```bash
cd server
cp .env.example .env      # заповни значення
npm install
npm start
```

Заповни `.env`:

| Змінна | Що це |
|---|---|
| `WFP_MERCHANT_ACCOUNT` | Merchant login з WayForPay |
| `WFP_MERCHANT_SECRET` | Merchant Secret Key з WayForPay |
| `WFP_MERCHANT_DOMAIN` | домен лендінгу без `https://` (напр. `chitkod.com.ua`) |
| `PRICE` | ціна (790) |
| `CURRENCY` | `UAH` |
| `BOT_USERNAME` | юзернейм бота без `@` |
| `LANDING_URL` | адреса, де лежить `index.html` |
| `SERVER_URL` | публічний URL цього сервера (напр. `https://pay.chitkod.com.ua`) |

> Файл `db.json` створюється автоматично (замовлення + токени). Для великого навантаження заміни просте JSON-сховище на Postgres/Redis.

## 2. Налаштування WayForPay

1. **Реквізити.** Кабінет → Налаштування магазину → Реквізити мерчанта → скопіюй **Merchant login** і **Merchant Secret Key** у `.env`.
2. **serviceUrl / returnUrl.** Наш сервер передає їх автоматично у запиті оплати:
   - `serviceUrl = SERVER_URL/callback`
   - `returnUrl  = SERVER_URL/return`
   Додатково впиши ці ж URL у налаштуваннях магазину WayForPay (розділ «Технічні налаштування / Повідомлення»), якщо потрібно за замовчуванням.
3. **POST на returnUrl.** У WayForPay (Повідомлення) переконайся, що дозволено надсилання даних на returnUrl/serviceUrl (не вимкнено).

---

## 3. Підключення лендінгу

У `index.html` у блоці `CONFIG`:

```js
paymentUrl: 'https://pay.your-domain.com/pay',   // ← URL цього сервера + /pay
```

Кнопки «Відкрити доступ» уже ведуть на `startPayment()`, який перекидає на `/pay`.
Повернення з `?status=approved&tg=...` обробляється автоматично: спрацьовує подія Facebook **Purchase** і редірект у бота за одноразовим посиланням.

---

## 4. Налаштування бота в SendPulse (видача курсу + перевірка токена)

1. **Тригер.** У флоу Telegram-бота зроби запуск за командою `/start` з параметром (deep-link `?start=<token>`). Значення параметра збережеться у змінну (напр. `start_param` / `#command_param`).
2. **HTTP-запит.** Додай елемент **«Запит»/«HTTP request»** (Send/receive data):
   - Метод: `GET`
   - URL: `https://pay.your-domain.com/validate?token={{start_param}}&uid={{contact.id}}`
     (замість `{{start_param}}` і `{{contact.id}}` підстав реальні змінні SendPulse: параметр запуску та id підписника)
   - Відповідь збережи у змінні: `ok` → з поля `ok`, `vstatus` → з поля `status`.
3. **Фільтр.** Додай елемент **«Фільтр»** за умовою `ok = true` (або `vstatus = valid`):
   - гілка **так** → повідомлення з видачею міні-курсу (посилання/файли/відео, за днями);
   - гілка **ні** → повідомлення «Це посилання вже використане або недійсне. Якщо ти оплатив(ла) — напиши в підтримку».
4. (Опційно) признач тег `paid_chitkod` у гілці успіху — зручно для сегментів і розсилок.

> Порада: у `/validate` ми привʼязуємо токен до першого `uid`. Тому той самий покупець може повернутися в бота повторно (отримає `valid`), а інша людина за пересланим посиланням — `used`. Якщо потрібна **строга** одноразовість (навіть для покупця — лише один вхід), не передавай `uid` у запиті: тоді токен згорає після першого відкриття.

---

## 5. Тестування

1. **Тестовий режим WayForPay** (до активації магазину) — платежі не списують реальні кошти.
2. Пройди шлях: лендінг → «Відкрити доступ» → оплата → повернення → редірект у бота.
3. У боті перевір, що перше відкриття дає доступ, а повторне з іншого акаунта — відмову.
4. Дивись статуси у WayForPay та у файлі `db.json` (orders/tokens).

## Ендпойнти сервера

| Метод | Шлях | Призначення |
|---|---|---|
| GET | `/pay` | створює замовлення й редіректить на WayForPay |
| POST | `/callback` | serviceUrl: перевірка підпису, підтвердження оплати, випуск токена |
| ALL | `/return` | returnUrl: сторінка очікування → редірект на лендінг з deep-link |
| GET | `/order-status?ref=` | статус замовлення для сторінки очікування |
| GET | `/validate?token=&uid=` | одноразова перевірка токена для SendPulse-бота |

## Безпека

- Секретний ключ WayForPay зберігається **лише на сервері** (в `.env`), ніколи не потрапляє в лендінг.
- Оплата підтверджується **підписом** у `/callback` (HMAC-MD5), а не параметром у браузері — підробити не можна.
- Одноразовість гарантується «спалюванням» токена у `/validate`.
